# Post-3.0.0 follow-up roadmap

**Date:** 2026-04-20
**Branch:** `feat/post-3.0-followups` (off F3-B final at `486d9a0`).

Index of the three follow-up tasks queued after 3.0.0 ships, with recommended interleaving.

## The three tasks

| # | Spec file | Target release | Scope |
|---|---|---|---|
| **1** | `2026-04-20-task1-plugin-registry-custom-endpoint-saml.md` | 3.1.0 | Async plugin factory registry + Custom endpoint monitor + Federated + Okta |
| **2** | `2026-04-20-task2-async-mysql-drivers.md` | 3.2.0 | `aiomysql` driver (asyncmy deferred to community request; asyncpg dropped) |
| **4** | `2026-04-20-task4-integration-tests-live-aurora.md` | 3.1.0+ (rolling) | Live-Aurora integration tests, extended per-task |

Task 3 (asyncpg) was dropped because asyncpg is not PEP 249-compliant — would require a separate DBAPI-adapter layer, materially more work than the other drivers, for a driver our users haven't requested. If demand surfaces later, revisit with a fresh spec.

## Recommended interleaving

```
Task 1-A (plugin registry)          ─┐
                                      ├─▶ 3.1.0 alpha
Task 4-A (integration baseline, PG) ─┘

Task 2 (aiomysql)                   ─┐
                                      ├─▶ 3.2.0
Task 4-C (MySQL integration tests)  ─┘

Task 1-B/C/D (custom endpoint +     ─┐
  federated + Okta auth)              ├─▶ 3.1.0 final  (or 3.3.0 depending on
Task 4-D (T1 scenarios in IT)       ─┘                  IdP test infra decisions)
```

Ship each slice together so its validation rides with it to production.

## Why this order

- **Plugin registry first**: unlocks string-based config (`plugins="failover,efm"`) for every async plugin shipped by F3-B. Largest user-facing win per unit of code. Doesn't depend on any driver work.
- **Integration baseline next**: validates F3-B's psycopg async path against live Aurora before we extend to more drivers. A failure here would block everything downstream, so catch it early.
- **aiomysql then**: extends the driver matrix to MySQL. First real validation of F3-B's master-spec invariant 8a — if any plugin needs to change when we add aiomysql, that's a bug in the `AsyncDriverDialect` ABC.
- **Custom endpoint + SAML/Okta last**: these need external infra (IdP or simulator). Delaying lets us gather real user demand signals and choose between "real IdP test" vs "mock IdP" for the integration layer with more information.

## Dropped from original 4-task list

- **asyncmy driver**: deferred to community request. aiomysql wins on maturity, SA-native support, pure Python (no Cython wheel lag), deployment ergonomics. The asyncmy driver-level perf advantage vanishes under Aurora network latency + wrapper plugin overhead. If a real workload surfaces where it matters, add as a second file under `aio/driver_dialect/asyncmy.py` — not a refactor.
- **asyncpg driver**: dropped. asyncpg is not PEP 249-compliant (no cursors, `$N` paramstyle, native transaction context manager). Would require a SA-adapter layer wrapping asyncpg to present a DBAPI shape. Materially more work than either MySQL driver, serving a use case our users haven't asked for. psycopg remains the default/only async PG driver.

## Out-of-scope escape hatches

Each task's own spec lists its escape hatches. Cross-task items:

- **Sync-path changes**: none of these three tasks modify the sync API. If one requires a sync change, escalate to its own spec.
- **New sync drivers**: not happening. Sync path stays on psycopg + mysql-connector-python.
- **Removing pre-3.0 APIs**: no. 3.x is strictly additive through 3.2.0.

## Parallel-work note

For a solo developer this remains **serial work, smart ordering**. The recommended interleaving above is the minimum viable "one-at-a-time" sequence that puts features with their validation. Don't attempt two tasks simultaneously.
